# 13 — Request for Quote (RFQ)

Status: **PARTIALLY IMPLEMENTED** — Database schema exists and is referenced by PurchaseOrders module, but standalone UI folder (`/RFQ/`) is not yet developed. The RFQ functionality is currently accessed through the Purchasing navigation hub and integrated into the PO workflow.

The RFQ module represents the procurement pre-order workflow: requesting quotes from suppliers before committing to a Purchase Order.

## 3. Integration Points

### 3.1 Generate PO from RFQ

**File**: `PurchaseOrders/GenerateFromRFQ.asp`

**URL**: `…/PurchaseOrders/GenerateFromRFQ.asp?RFQid=<n>`

**Logic**:
```asp
1. Load RFQ header by RFQid
2. Build query string with:
   - Code, ContactId, DivisionId, PODate = RFQDate
   - POStatusId = 1 (Draft)
   - PriceExTotal = RFQ.TotalEx
   - PriceIncTotal = RFQ.TotalInc
3. Loop RFQContents:
   - Append Quantity, Description, PriceEx, PriceExSubTotal
4. URL-encode and redirect to:
   PurchaseOrders/Add2.asp?<query_string>
```

**Behavior**: The PO Add form opens with all RFQ data pre-filled. User can modify before saving.

### 3.2 RFQ Reference in PO

When a PO is generated from an RFQ:
- `PurchaseOrders.RFQid` is set to the source RFQ ID
- `ViewHistory.asp` shows the RFQ origin with supplier name
- `Add_Proc.asp` writes audit: `"Generated by RFQ #<n>"`

## 4. Access Control

The Purchasing navigation hub checks for RFQ access:
```asp
hasRFQAccess = (Request.Cookies("DivisionIdsAccess")("RFQ") <> "0")
```

This is distinct from PurchaseOrders access - a user might have access to view RFQs but not create POs, or vice versa.

## 6. Relationship to Quotes

The RFQ table has a `Qid` field (like PurchaseOrders), suggesting RFQs can be related to Quotes:
- A Quote might require purchasing goods from suppliers
- RFQ workflow gets quotes from suppliers
- Selected supplier quote converts to PO
- PO links back to original Quote

This represents the full procurement chain:
```
Customer Quote → RFQ to Suppliers → Select Supplier → PO to Supplier → Invoice Customer
```

## 7. Known Limitations

1. **No Standalone UI**: The RFQ folder doesn't exist in the codebase despite being linked from Purchasing navigation
2. **No Status Workflow**: Unlike POs, there's no visible approval/escalation workflow for RFQs
3. **No PDF Generation**: No evidence of RFQ PDF generation handlers (would mirror PO pattern)
4. **No Email Integration**: No RFQ-specific email sending logic found

---

## 8. Usage in Current System

Currently, RFQ data is used in these ways:
1. **Origin Tracking**: PO ViewHistory shows when a PO originated from an RFQ
2. **Data Pre-fill**: RFQ data accelerates PO creation
3. **Audit Trail**: `PurchaseOrderAudit` records RFQ origin
